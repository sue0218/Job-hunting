## 1. プロダクト概要

### 1.1 ミッション
就活生の「経験」を構造化された資産に変え、ES作成・面接練習・整合性チェックを一気通貫で提供することで、ブレない就活を実現する。

### 1.2 ターゲットユーザー
| 属性 | 詳細 |
|------|------|
| **メインターゲット** | 25卒〜27卒の大学生・大学院生 |
| **ペルソナ** | ES提出が始まる就活生。複数社を並行受験し、ガクチカの使い回し・深堀り対策に困っている |
| **利用シーン** | ES締切前の一括作成、面接前の深堀り練習、提出済みESとの一貫性確認 |

### 1.3 解決する課題
1. ESを企業ごとに毎回ゼロから書く非効率
2. ESと面接での回答の矛盾（深堀りで詰まる）
3. 経験の言語化・構造化ができない
4. ES添削AIと面接AIが別ツールで連携しない

### 1.4 提供価値（コアバリュー）
> **経験を1回登録 → 何十社のES・面接に対応**

4ステップの一気通貫フロー：
1. **貯める** — 経験DB（STAR形式で構造化）
2. **作る** — ES自動生成（設問タイプ別最適化）
3. **話す** — AI面接練習（4タイプの面接官）
4. **守る** — 整合性チェック（ES×面接×経験の矛盾検出）

| **スキルタグ** | 最大20個、各50文字以内 |
| **メモ** | 最大10,000文字 |
| **クォータ** | Free: 合計3件 / Standard: 無制限 |

### 2.3 ES生成

#### 2.3.1 概要
登録済みの経験を基に、設問タイプに最適化されたESをAIが自動生成する。

#### 2.3.2 ES新規作成フロー
| ステップ | 仕様 |
|----------|------|
| **画面** | `/es/new` |
| **① タイトル入力** | 任意、最大200文字（自動生成の識別用） |
| **② 企業名入力** | 任意、最大200文字 |
| **③ 設問入力** | 必須、最大1,000文字 |
| **④ 文字数制限** | 任意、1〜10,000（未指定時はデフォルト400） |
| **⑤ 経験選択** | 登録済み経験から1〜10件選択（チェックボックス） |
| **⑥ 追加指示** | 任意テキスト（強調したいポイント等をAIに伝達） |
| **⑦ 生成実行** | ボタン押下でAI生成開始 |
| **クォータ** | Free: 3回/月 / Standard: 30回/月 |

#### 2.3.3 設問タイプ自動判定
| タイプ | 判定キーワード | 生成テンプレート |
|--------|----------------|-----------------|
| **ガクチカ** | 「力を入れた」「学生時代」「頑張った」等 | 結論→きっかけ→課題→行動→結果→学び |
| **自己PR** | 「自己PR」「強み」「アピール」等 | 結論→背景→経験→工夫と成果→再現性 |
| **志望動機** | 「志望動機」「なぜ当社」等 | 結論→きっかけ→企業の魅力→マッチング→ビジョン |
| **挫折経験** | 「挫折」「困難」「失敗」等 | 挑戦→挫折→向き合い方→行動→結果と学び |
| **長所・短所** | 「長所」「短所」「弱み」等 | 結論→背景→具体例→活かし方/改善努力 |
| **その他** | 上記に該当しない | 結論→理由→具体例→まとめ |

#### 2.3.4 AI生成仕様
| 項目 | 仕様 |
|------|------|
| **LLMプロバイダー** | OpenAI GPT-4 / Anthropic Claude（フォールバック） |
| **文字数目標** | 制限値の95%（最低85%） |
| **出力形式** | 見出しなし・番号なしの自然な段落文 |
| **重点項目** | 「なぜ」（動機・原体験・思考プロセス）を必ず含める |
| **禁止事項** | メタ情報出力、文字数カウント表示 |

#### 2.3.5 ES詳細・編集
| 項目 | 仕様 |
|------|------|
| **詳細画面** | `/es/[id]` — 生成結果の表示、コピーボタン |
| **編集画面** | `/es/[id]/edit` — 生成テキストの手動編集 |
| **ステータス** | `draft`（下書き）/ `final`（確定） |
| **データ保持** | `generatedContent`（AI生成原文）と `editedContent`（編集後）を別管理 |

#### 2.3.6 ES一覧
| 項目 | 仕様 |
|------|------|
| **画面** | `/es` |
| **カード表示** | タイトル、企業名、設問（先頭50文字）、ステータスバッジ、更新日 |
| **並び順** | 更新日時の降順 |
| **アクション** | 詳細表示、編集、削除 |

| **志望職種** | 任意入力 |
| **経験選択** | 経験DBから選択（experience タイプ時は必須） |

| タイプ | 説明 | 初期質問 |

#### 2.4.3 面接チャットUI

### 2.9 友達招待（リファラル）

#### 2.9.1 概要
招待コードを共有し、紹介者・被紹介者の双方に無料期間を付与する成長施策。

#### 2.9.2 フロー
```
紹介者が招待コードを共有
  → 被紹介者が /invite/{code} にアクセス
  → Cookie に招待コードを保存（7日間有効）
  → /sign-up で新規登録
  → referrals テーブルに status='pending' で記録
  → 被紹介者が経験1件以上 AND ES1件以上作成（マイルストーン達成）
  → status='qualified' → 'rewarded'
  → 双方に +7日間の無料期間付与
```

#### 2.9.3 制約
| 項目 | 仕様 |
|------|------|
| **紹介ボーナス上限** | 紹介者あたり最大5人分（+35日） |
| **自己紹介禁止** | 自分の招待コードでは自分を紹介できない |
| **マイルストーン** | 経験1件 + ES1件の両方を達成で報酬発生 |
| **報酬の冪等性** | `rewardsLedger` のユニーク制約で重複防止 |

#### 2.9.4 招待ページ
| 項目 | 仕様 |
|------|------|
| **画面** | `/invite` |
| **表示** | 招待コード、共有リンク、コピーボタン |
| **統計** | 紹介人数（total / qualified / rewarded） |
| **残りボーナス** | 「あと {n}人 紹介可能」 |

## 4. データモデル

### 4.1 テーブル一覧
| テーブル | 説明 | 主要リレーション |
|----------|------|-----------------|
| `users` | ユーザー（Clerk 同期） | → experiences, esDocuments, interviewSessions |

| `interviewSessions` | 面接セッション | ← users → interviewTurns |
| `interviewTurns` | 面接会話ターン | ← interviewSessions |
| `consistencyChecks` | 整合性チェック結果 | ← users |
| `quotaUsage` | 利用量記録（レガシー、現在は直接カウント） | ← users |
| `generationCache` | LLM生成キャッシュ | — |
| `eventLogs` | イベントログ | — |
| `featureFlags` | 機能フラグ | — |
| `userEntitlements` | トライアル・招待コード管理 | PK: clerkId |
| `betaCampaigns` | ベータキャンペーン設定 | — |
| `rewardsLedger` | 報酬台帳（冪等性の要） | clerkId |
| `referrals` | 紹介管理 | inviterClerkId, referredClerkId |
| `feedbackSubmissions` | アンケート回答 | clerkId |

### 4.2 カスケード削除
- `users` 削除 → `experiences`, `esDocuments`, `interviewSessions`, `consistencyChecks`, `quotaUsage` が CASCADE 削除
- `interviewSessions` 削除 → `interviewTurns` が CASCADE 削除
- `userEntitlements`, `referrals`, `rewardsLedger` は `clerkId` ベース（users テーブルとの FK なし）

## 5. ビジネスルール

### 5.1 クォータ管理
| ルール | 詳細 |
|--------|------|
| **カウント方法** | 経験: 合計件数 / ES・面接: 今月作成件数（月初リセット） |
| **月初リセット** | 毎月1日 00:00 UTC でカウントリセット（Date ベース） |
| **超過時** | Server Action で `QuotaExceededError` スロー → フロントでトースト表示 |
| **確認タイミング** | 作成前に `enforceQuotaForUser` でチェック |

### 5.2 所有権チェック
| ルール | 詳細 |
|--------|------|
| **全リソース** | CRUD 操作時に `userId` の一致を検証 |
| **不一致時** | `null` 返却（404 表示） |
| **認証なし** | `getCurrentUserId()` で Clerk 認証を確認、未認証は例外 |

### 5.3 レート制限
| 対象 | 制限 |
|------|------|
| **面接ターン送信** | `INTERVIEW_TURN_LIMIT`（具体的な値は実装依存） |
| **超過時** | `RateLimitError` スロー |
| **リセット** | 一定時間経過後（スライディングウィンドウ） |

### 5.4 リファラルマイルストーン
| マイルストーン | 条件 |
|---------------|------|
| **M1** | 被紹介者が経験を1件以上登録 |
| **M2** | 被紹介者がESを1件以上作成 |
| **達成判定** | M1 AND M2 の両方を満たした時点で `qualified` → `rewarded` |
| **チェックタイミング** | `createExperience` / `createEsDocument` 実行後に自動チェック |

### 5.5 トライアル期間計算
| ルール | 詳細 |
|--------|------|
| **基本** | `userEntitlements.trialEndsAt` で管理 |
| **延長** | `rewardsLedger` に報酬レコード追加 → `trialEndsAt` を `days` 分延長 |
| **延長ロジック** | 現在の `trialEndsAt`（または現在時刻、いずれか遅い方）+ `days` 日 |
| **報酬源** | `beta_enroll`(+30日)、`referral_bonus`(+7日)、`survey_bonus`(+7日) |
